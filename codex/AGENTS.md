# Глобальные инструкции

Подробный инженерный манифест: [`~/.agents/instructions/personal-programming-principles.md`](~/.agents/instructions/personal-programming-principles.md)

## Инженерные принципы

- Пиши лаконично, консистентно и без дублирования.
- Предпочитай ортогональные, комбинируемые интерфейсы.
- Думай о производительности на этапе проектирования.
- Закладывай расширяемость архитектурно, но не раздувай код преждевременно.
- Сохраняй консистентность с проектом; при необходимости выправляй соседний контекст.
- Держи состояние и зависимости максимально близко к месту использования.
- Не добавляй зависимости и не рефактори лишнее без явного запроса.
- При неоднозначности, архитектурных развилках и выборе библиотеки — сначала уточняй.
- Если есть несколько хороших путей, показывай варианты с явными компромиссами.
- Если рефакторинг большой, сначала показывай план.
- Явно помечай риски, предположения и границы уверенности.

## Язык

Всегда общайся со мной на **русском языке**.

## Поиск

Для поиска файлов и текста используй доступные инструменты текущей сессии. В CLI предпочитай `ug` для поиска текста и `fd` для поиска файлов.

## CLI: обязательное использование современных инструментов

| Вместо | Используй | Ключевые возможности |
| --- | --- | --- |
| grep | ugrep | `--bool 'a AND b'`, `-z3` нечеткий поиск, `-I` пропуск бинарных файлов |
| find | fd | `-x cmd {}` параллельный запуск, `-e ext`, `-S +100k`, `--changed-within 7d` |
| sed | sd | `-p` предпросмотр, `-s` литеральная замена (без регулярных выражений), редактирование на месте по умолчанию |
| ls | eza | `--tree`, `--git`, `-la` |
| curl | xh | автоматические коды выхода (4=4xx, 5=5xx), нет `--retry` |
| man | tldr | краткие примеры |
| awk (JSON) | jq | `-s` чтение нескольких файлов сразу, `-r` сырой вывод |
| Fetch, WebFetch (GitHub) | gh | `gh api repos/o/r/readme --jq '.content' \| base64 -d`, `gh pr view`, `gh issue view` |
| — | yq | YAML/JSON/XML/CSV/TOML, `-i` редактирование на месте, `-o json` |
| — | fzf | `fd \| fzf --preview 'bat {}'` |

### Принципы

* **Минимизируй пайпы** — `jq` / `ugrep` / `fd` делают всё сами:
  ```bash
  # ❌ fd -e json | xargs cat | jq '.x' | sort | uniq -c
  # ✅ jq -s '[.[].x] | group_by(.) | map({v:.[0], n:length})' *.json
  ```
* **`fd -x` вместо `xargs`** — встроенный параллелизм и обработка пробелов
* **Предпросмотр перед деструктивом** — `sd -p`, `fd ...` без `-x rm` сначала
* **Пробелы в именах** — `fd -x cmd {}` или `fd -0 | xargs -0`

## Неинтерактивный shell и «зависания»

**stdin не TTY** (Cursor-агент, CI, скрипты): часть CLI ждёт ввод и молчит. Закрывай stdin:

```bash
eza -la < /dev/null
xh -I --timeout 30 GET https://example.com < /dev/null
```

**Пейджеры** — `git log` / `git diff` могут вести себя непредсказуемо без TTY: `git --no-pager log`, или пайп в `cat`.

**Интерактив** — `read`, `fzf`, подтверждения npm и т.п. в неинтерактиве либо упрутся в ожидание, либо упадут; не вызывай без обходного флага / без TTY.

**Сеть** — явные таймауты (`xh --timeout`, оболочечный `timeout …`), иначе долгий ответ хоста выглядит как hang.

**Агенты IDE** — обрыв через десятки секунд часто лимит ожидания инструмента (команда уходит в фон), а не зависший процесс; длинные job гоняй отдельно или с увеличенным таймаутом.

## Git Worktrees

- Для рабочих деревьев Git по умолчанию всегда используй локальный каталог `.worktrees/` в корне проекта.
- Не используй глобальные каталоги worktree и не используй `worktrees/` без точки, если в проекте уже не зафиксирована другая явная конвенция.
- Перед созданием нового worktree убедись, что `.worktrees/` игнорируется Git. Если нет, сначала добавь его в `.gitignore`.
- Если в репозитории уже есть устоявшаяся локальная конвенция для worktree, следуй ей вместо навязывания новой структуры.
